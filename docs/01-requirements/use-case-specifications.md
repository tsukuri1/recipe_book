# Use Case спецификация

![Use Case Diagram](../images/01-use-case-diagram.png)

| Use Case | Актер | Основной сценарий |
|---|---|---|
| Создать рецепт | Пользователь | Открыть форму, заполнить поля, сохранить |
| Добавить в покупки | Пользователь | Открыть рецепт, нажать кнопку добавления |
| Одобрить рецепт | Администратор | Открыть админ-панель, нажать «Одобрить» |
| Изменить профиль | Пользователь | Открыть профиль, выбрать аватар, сохранить имя |

## UC-01. Создать рецепт

| Поле | Описание |
|---|---|
| Актор | Пользователь |
| Предусловие | Пользователь авторизован |
| Основной сценарий | Открыть экран добавления, заполнить поля, выбрать изображение, сохранить |
| Постусловие | Рецепт создан со статусом `PENDING` |
| Альтернатива | Если поля пустые, приложение подставляет безопасные значения по умолчанию |

## UC-02. Добавить ингредиенты в покупки

Пользователь открывает карточку рецепта и нажимает кнопку добавления ингредиентов. Система создаёт позиции списка покупок на основе ингредиентов рецепта. После этого пользователь может отмечать позиции как купленные.

## UC-03. Одобрить рецепт

Администратор открывает админ-панель, просматривает рецепты на проверке и выбирает действие «Одобрить». Backend меняет статус рецепта на `APPROVED`, после чего рецепт может отображаться в общем каталоге.

## UC-04. Изменить профиль

Пользователь открывает профиль, вводит новое имя или выбирает аватар. Данные сохраняются в `ProfileStore` с привязкой к email, чтобы у разных аккаунтов были разные профили.
