## Подход к требованиям

Требования сформированы от пользовательских сценариев. Сначала выделены акторы: обычный пользователь и администратор. Затем для каждого актора определены цели: найти рецепт, создать рецепт, добавить ингредиенты в покупки, изменить профиль, обработать жалобу или модерировать рецепт.

![Use Case Diagram](../images/01-use-case-diagram.png)

![Domain Model](../images/02-domain-model.png)

Для мобильной траектории особенно важны нефункциональные требования: приложение должно иметь понятную навигацию, обрабатывать состояния загрузки и ошибки, использовать REST API и поддерживать локальное хранение данных. Поэтому требования связаны не только с экранами, но и с архитектурными слоями Android-клиента и backend.

## Использование раздела

Раздел используется как основа для проектирования: Use Case диаграмма показывает границы системы, Domain Model помогает выделить сущности, а трассировка показывает, где каждое требование реализовано в коде и документации.
