# Data Mapper

![DTO Mapping Diagram](../images/17-dto-mapping-diagram.png)

DTO и локальные сущности преобразуются в доменные модели через `foundation/DtoMappers.kt`. Это отделяет формат API/Room от UI и бизнес-логики.

## Зачем нужен Data Mapper

Формат данных на сервере, в локальной базе и в интерфейсе отличается. Например, REST API может возвращать DTO с полями, удобными для JSON, Room хранит данные в виде cache entity, а UI работает с доменной моделью `Recipe`, где уже есть готовые списки ингредиентов, шагов и КБЖУ.

Если смешать эти форматы, UI начнёт зависеть от структуры API, а изменения backend будут ломать экраны. Data Mapper решает эту проблему: каждое преобразование находится в одном месте.

## Примеры преобразований

| Откуда | Куда | Назначение |
|---|---|---|
| `RecipeDto` | `Recipe` | Показ рецепта в UI |
| `Recipe` | `CreateRecipeRequest` | Отправка нового рецепта на сервер |
| `RecipeCacheEntity` | `Recipe` | Восстановление рецепта из Room |
| `UserSettingsDto` | `UserSettings` | Отображение настроек |

## Вывод

Data Mapper снижает связанность между слоями Foundation и Presentation. Это особенно важно для мобильного приложения, где источник данных может переключаться между сервером и локальным кэшем.
