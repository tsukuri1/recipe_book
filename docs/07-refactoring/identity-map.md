# Identity Map

![Local Storage Diagram](../images/16-local-storage-diagram.png)

В Android-клиенте роль Identity Map частично выполняет репозиторий и локальный Room-кэш: повторные обращения к рецептам синхронизируются через единый источник данных.

| Область | Решение |
|---|---|
| Рецепты | Кэширование в Room через `RecipeCacheEntity` |
| Покупки | Кэширование состояния `ShoppingCacheEntity` |
| Профиль | `ProfileStore` хранит данные по email пользователя |

## Как это работает в проекте

Классический Identity Map гарантирует, что один и тот же объект в рамках сессии представлен одной и той же экземплярной сущностью. В мобильном учебном проекте эта идея адаптирована через единый источник данных: репозиторий и локальный кэш не позволяют экранам самостоятельно создавать независимые версии одного рецепта.

Когда каталог загружается с сервера, `CachedRecipeRepository` сохраняет рецепты в Room. Если сеть недоступна, данные восстанавливаются из локального кэша. Так пользователь видит согласованную версию каталога, а не набор разрозненных списков.

## Ограничения

Текущая реализация не решает сложные конфликты синхронизации, например одновременное редактирование одного рецепта на разных устройствах. Для курсовой мобильной траектории достаточно показать идею единого источника данных и offline fallback.

## Возможное развитие

В промышленной версии можно добавить полноценную синхронизацию с версиями записей, временем последнего изменения и политикой разрешения конфликтов. Тогда Identity Map будет работать вместе с Unit of Work и позволит безопасно синхронизировать изменения между несколькими устройствами.

## Вывод

Для текущего проекта Identity Map представлен как архитектурный принцип согласованности данных: экраны получают данные через единый репозиторий, а локальный кэш не конкурирует с UI, а обслуживает его.
