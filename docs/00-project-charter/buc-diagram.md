# BUC диаграмма

![Use Case Diagram](../images/01-use-case-diagram.png)

## Бизнес-use cases

| Актор | Бизнес-сценарий | Результат |
|---|---|---|
| Пользователь | Найти рецепт | Подобран рецепт по названию или категории |
| Пользователь | Добавить собственный рецепт | Рецепт отправлен на модерацию |
| Пользователь | Сформировать список покупок | Ингредиенты добавлены в чек-лист |
| Пользователь | Настроить профиль | Имя и аватар сохранены для аккаунта |
| Администратор | Проверить рецепт | Рецепт одобрен или отклонён |
| Администратор | Рассмотреть жалобу | Жалоба переведена в работу или закрыта |

## Описание бизнес-процесса

Бизнес-процесс начинается с потребности пользователя приготовить блюдо. Пользователь открывает каталог, выбирает категорию или вводит поисковый запрос, затем просматривает карточку рецепта. Если блюдо подходит, ингредиенты могут быть перенесены в список покупок. После покупки продуктов пользователь отмечает позиции как выполненные.

Второй важный процесс связан с пополнением каталога. Пользователь создаёт рецепт, указывает название, категорию, ингредиенты, шаги приготовления, КБЖУ и фото. Такой рецепт не сразу попадает в общий каталог: он получает статус `PENDING` и попадает в очередь администратора. Администратор проверяет корректность данных, после чего одобряет или отклоняет запись.

## Основной поток

1. Пользователь авторизуется в приложении.
2. Открывает каталог рецептов.
3. Находит блюдо через поиск или категорию.
4. Открывает карточку рецепта.
5. Добавляет ингредиенты в список покупок.
6. Отмечает купленные продукты.

## Альтернативные потоки

Если нужного рецепта нет, пользователь создаёт собственный рецепт. Если рецепт содержит ошибку, пользователь или администратор фиксирует проблему через жалобу. Если соединение с сервером недоступно, приложение использует локальный кэш для уже загруженных данных.
