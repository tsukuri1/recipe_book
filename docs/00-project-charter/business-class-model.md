# Модель бизнес-классов

![Domain Model](../images/02-domain-model.png)

| Класс | Назначение | Основные атрибуты |
|---|---|---|
| User | Пользователь приложения | name, email, role, avatarUri |
| Recipe | Кулинарный рецепт | title, category, ingredients, steps, nutrition, status |
| Ingredient | Ингредиент рецепта | name, amount, unit |
| CookingStep | Шаг приготовления | order, description |
| ShoppingItem | Позиция списка покупок | name, checked |
| UserSettings | Пользовательские настройки | syncEnabled, offlineCacheEnabled, notificationsEnabled |
| Complaint | Жалоба на рецепт | title, description, status |

## Связи между бизнес-классами

`User` является центральной сущностью пользовательского сценария: он создаёт рецепты, добавляет блюда в избранное, формирует список покупок и настраивает профиль. `Recipe` агрегирует ингредиенты, шаги приготовления и пищевую ценность. Такая структура позволяет показывать рецепт как полноценную карточку, а не как простой текстовый список.

`ShoppingItem` связан с рецептом косвенно: позиции списка покупок создаются из ингредиентов выбранного рецепта. Это удобно для пользователя, потому что ему не нужно вручную переписывать продукты. `Complaint` и административные статусы нужны для контроля качества общего каталога.

## Бизнес-правила

- Рецепт, созданный пользователем, сначала получает статус `PENDING`.
- В общий каталог попадают только рецепты со статусом `APPROVED`.
- Администратор может менять статус рецепта и обрабатывать жалобы.
- Профиль и аватар пользователя должны сохраняться отдельно для каждого email.
- Список покупок должен поддерживать два состояния позиции: купить и куплено.

## Вывод

Модель бизнес-классов покрывает основные процессы приложения: поиск рецепта, создание рецепта, модерацию, покупки, профиль и настройки. Набор сущностей достаточен для мобильной траектории и соответствует требованию методички о предметной области с несколькими связанными объектами.
